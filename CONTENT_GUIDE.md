# Content guide — run every case study through this before publishing

The layout enforces the structure; this checklist enforces the substance. If a
write-up passes every item, it's ready. If it fails one, fix the content — not
the checklist.

## The 30-second test

- [ ] **Does the TL;DR stand alone?** Cover the hook, cover image, and everything below "The short version." A hiring manager who reads *only* the TL;DR should know: the problem, what you did, and how it turned out.
- [ ] **Is the hook outcome-oriented?** "Kitchen feedback reimagined so essential moments can be seen" — not "I designed a kitchen device." If the hook starts with "I designed/created/made," rewrite it around what changed for the user.

## Ownership

- [ ] **Is your role unambiguous?** The metadata bar says what *you* did. On team projects, name your slice: "I led play-testing and designed the scrap system" — never just "team of 3."
- [ ] **First person, active voice throughout.** "I interviewed," "I chose," "I scrapped." If a sentence hides behind "it was decided" or "the team explored," claim it or attribute it.

## Evidence (show, don't tell)

- [ ] **Every claim has receipts.** "I'm passionate about research" → delete. "I interviewed 8 line cooks and found…" → keep. Search your draft for *passionate, love, believe, always* — each one is usually a claim wearing a costume.
- [ ] **All scaffold numbers are gone.** The placeholder copy shipped with invented specifics (participant counts, quotes, findings). Every number, quote, and finding on the page must now be yours and true. Don't round up. Honest small numbers beat impressive fake ones.
- [ ] **The problem is grounded in something real** — an observation, an interview insight, a statistic — and it appears *before* any solution does.

## Images and captions

- [ ] **Every image earns its place.** If you can delete an image and the story doesn't lose a step, delete it.
- [ ] **Every caption explains why, not what.** "Early sketches" fails. "Round-two sketches — flattening the base here is what later solved the tipping problem" passes. The caption's job is the artifact's role in the story.
- [ ] **At least one failure is visible.** A dead-end direction, a prototype that flunked testing. Iteration without failure reads as fiction.
- [ ] **Alt text describes the image** (for screen readers); the caption interprets it. They're different jobs — don't duplicate.

## Decisions

- [ ] **2–3 key decisions, each with a real trade-off.** "I chose X" is a fact; "I chose X and it cost me Y, which I accepted because Z" is a skill. If a decision has no cost, it wasn't a decision.

## Outcome

- [ ] **Results are honest and specific.** Testing feedback, metrics, exhibition, adoption — whatever actually happened. "What I'd do next" is a valid result; vague success is not.
- [ ] **"What I learned" is specific to this project.** If the lesson could be pasted into any other case study ("I learned the value of iteration"), it's not a lesson yet. It should name the moment that taught it.

## Fluff sweep (last pass)

- [ ] No "in today's fast-paced world" openers — or any opener that could begin someone else's case study.
- [ ] No "seamless," "delightful," "innovative," "leverage" without evidence attached.
- [ ] Read the whole thing aloud once. Anywhere you cringe or skim, cut.

## Mechanics

- [ ] TL;DR is 3–4 sentences, not a paragraph stack.
- [ ] Hook is one sentence.
- [ ] `order` field set so the grid tells the story you want (strongest work first).
- [ ] All placeholder `.svg` images replaced (search the file for `.svg`).
- [ ] Real `alt` text on every image.
